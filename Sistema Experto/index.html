¿Por qué Diabetes Tipo 2?

Se seleccionó la diabetes por múltiples razones: alta prevalencia (14.1% de adultos), síntomas claramente identificables por el paciente, alto impacto preventivo — el diagnóstico temprano puede retrasar o evitar complicaciones graves (ceguera, amputaciones, insuficiencia renal), y por la factibilidad de modelar su base de conocimiento con reglas if-then bien establecidas médicamente.

Síntomas Clave del Sistema

El sistema evalúa 15 indicadores clasificados en tres categorías: 5 síntomas cardinales (peso 2-3), que son los más diagnósticos; 5 síntomas secundarios (peso 2), que refuerzan el diagnóstico; y 5 factores de riesgo (peso 1-2), que contextualizan el cuadro clínico. La tríada clásica (poliuria + polidipsia + polifagia) activa la regla de mayor riesgo.

Análisis del Sistema Experto

1. Base de Conocimiento

Implementada con diccionarios y listas Python. Cada síntoma es un diccionario con: código (ID único), nombre legible, descripción, peso (valor diagnóstico 1-3) y categoría. La base de conocimiento refleja criterios clínicos basados en las guías de la ADA y la NOM-015-SSA2 de México.

2. Motor de Inferencia IF-THEN

El motor evalúa 7 reglas en cascada:

• Regla 1: Clasificar síntomas y acumular score ponderado
• Regla 2: ¿Tríada clásica completa? → Riesgo ALTO
• Regla 3: ¿Pérdida de peso + 2 cardinales? → Riesgo ALTO
• Regla 4: ¿Signos neurológicos? → Neuropatía probable
• Regla 5: Determinar nivel por score total y umbrales
• Regla 6: Generar recomendaciones contextuales
• Regla 7: Construir explicación del razonamiento

3. Umbrales de Decisión

El sistema calcula un porcentaje de score sobre el máximo posible:

• ≥ 60% o tríada/pérdida de peso → Riesgo ALTO
• ≥ 35% o 2+ cardinales → Riesgo MODERADO
• ≥ 15% o 2+ síntomas → Riesgo BAJO
• < 15% → Riesgo Muy Bajo

4. Generación de Decisiones

Cada recomendación se genera por activación de reglas específicas. Si se detectan antecedentes familiares → pruebas semestrales. Si hay neuropatía probable → urgencia especial. El sistema explica por qué tomó cada decisión, a diferencia de un sistema de caja negra.

5. API REST con Flask

Endpoints disponibles:

• POST /api/evaluar — Recibe JSON con lista de síntomas
• GET /api/sintomas — Lista todos los síntomas disponibles
• GET /api/historial — Historial de evaluaciones
• GET /api/estadisticas — Métricas del sistema
• GET /api/esquema_sql — Esquema PostgreSQL

6. Limitaciones del Sistema

• No es diagnóstico definitivo — requiere confirmación médica
• Síntomas subjetivos — la percepción del paciente varía
• Sin datos numéricos — no incorpora glucosa medida ni IMC real
• Sin comorbilidades — no considera otras enfermedades
• Reglas fijas — no aprende de nuevos casos
• Sin contexto temporal — no evalúa duración de síntomas

Tabla: enfermedades

Catálogo de enfermedades detectables. Incluye nombre, descripción, prevalencia en México y fuentes bibliográficas. Permite escalar el sistema a múltiples enfermedades.

Tabla: sintomas

Contiene cada síntoma con código único, nombre, peso diagnóstico y categoría. Referencia a enfermedades mediante FK. Permite modificar síntomas sin tocar el código.

Tabla: evaluaciones

Historial completo de evaluaciones. Guarda síntomas ingresados (JSONB), nivel de riesgo, score y resultado completo para análisis retrospectivo.
